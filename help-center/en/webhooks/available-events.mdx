This is the list of events you can subscribe to when [setting up a webhook](/help-center/en/webhooks/setting-up-webhooks), with the content sent in each one.

## Delivery format

Every event is sent via **POST** with a JSON body in this format:

```json
{
  "id": "evt_01hj8q1k2t3r5v7x9z2b4d6f_12",
  "event": "payment.approved",
  "created_at": "2026-06-16T12:00:00Z",
  "data": { }
}
```

| Field | What it is |
|---|---|
| **id** | Unique identifier of the delivery (`evt_...`). Use it to [avoid processing duplicates](/help-center/en/webhooks/security-and-deliveries#idempotency). |
| **event** | Name of the event (see the table below). |
| **created_at** | Date/time of the event (ISO 8601). |
| **data** | Event-specific data. |

## The `user` object

Most events include a `user` object with the same structure, built for automations (including WhatsApp messaging from `phone`):

```json
{
  "id": 123,
  "email": "john@email.com",
  "name": "John",
  "phone": "11999998888",
  "phone_country": "BR"
}
```

- `phone` / `phone_country` come from the student's profile and may be `null` if they didn't provide a phone number.
- On checkout events before the account exists (`payment.pending`, `payment.failed`), `user` is resolved by email and may be `null` for an anonymous lead â€” in that case use `customer_email`.

## The class context (`class_id` / `class_name`)

Events tied to a class (cohort) carry `class_id` and `class_name`, saving you a lookup on your side. `class_name` is `null` when there is no associated class.

## List of events

| Event | Category | When it fires |
|---|---|---|
| `payment.approved` | Payments | Single payment approved. |
| `payment.pending` | Payments | Payment awaiting confirmation (PIX/boleto generated). |
| `payment.failed` | Payments | Payment declined or errored. |
| `payment.chargeback` | Payments | Chargeback received (dispute at the acquirer). |
| `subscription.created` | Subscriptions | First subscription charge approved. |
| `subscription.updated` | Subscriptions | Subscription status change. |
| `subscription.renewed` | Subscriptions | Subscription renewal approved (later cycles). |
| `subscription.past_due` | Subscriptions | Subscription went past due (unpaid invoice). |
| `subscription.payment_failed` | Subscriptions | A renewal invoice charge failed. |
| `subscription.canceled` | Subscriptions | Subscription canceled. |
| `student.enrolled` | Access | New access granted to a student. |
| `access.revoked` | Access | A student's access revoked. |
| `enrollment.expired` | Access | Access reached its expiry and was deactivated automatically. |
| `refund.created` | Refunds | Refund processed. |
| `certificate.issued` | Certificates | Certificate issued to a student. |
| `waitlist.joined` | Waitlist | Lead joined a class waitlist. |
| `waitlist.promoted` | Waitlist | Lead promoted from the waitlist (seat freed). |
| `waitlist.expired` | Waitlist | Waitlist promotion window expired. |
| `lesson.completed` | Engagement | Student completed a lesson. |
| `course.completed` | Engagement | Student completed 100% of the course lessons. |
| `assignment.submitted` | Engagement | Student submitted an assignment. |
| `assignment.reviewed` | Engagement | Assignment reviewed (approved, needs revision, or rejected). |
| `community.post_created` | Engagement | New community post. |

## Content (`data`) of each event

In the examples below, the `user` object follows the [standard format](#the-user-object).

### `payment.approved`

```json
{
  "payment_id": "...",
  "user": {
    "id": 123,
    "email": "john@email.com",
    "name": "John",
    "phone": "11999998888",
    "phone_country": "BR",
    "class_id": 45,
    "class_name": "Cohort 2026"
  },
  "amount": 497.00,
  "currency": "BRL",
  "gateway": "pagarme"
}
```

### `payment.pending`

```json
{
  "gateway_payment_id": "...",
  "checkout_id": "...",
  "customer_email": "john@email.com",
  "user": null,
  "amount": 497.00,
  "currency": "BRL",
  "payment_method": "pix",
  "gateway": "pagarme"
}
```

`user` carries the standard object when the email already has an account; `null` for an anonymous lead.
